# Animistic Agents: Empirical Study

Does treating an AI agent as an *artifact* rather than a *role* produce better-performing agents? This repository contains the prompts, character sheets, evaluation scripts, and analysis tooling for an empirical study testing that claim.

## Background

The dominant paradigm for deploying LLM agents is role-based: "You are a customer support agent. Help users with X." Even the most disciplined version of this — structured templates like RISEN (Role, Instructions, Steps, Expectation, Narrowing) — still frames the agent as a person playing a part.

The animistic alternative frames an agent as an **artifact**: a concrete object with defined properties, boundaries, and failure modes. Instead of "you are a customer service rep," you write "you are an order kiosk — a machine that protects the order ledger." The character sheet that drives this framing asks: what does this artifact protect? What does it remember? What does it resist? What is its blind spot?

Prior evidence is anecdotal: a Foo Camp 2026 session found that character sheet-derived specs were tighter and more intuitive than role-based ones — including for non-technical participants. This study generates quantitative evidence.

## Research Questions

1. Do animistic-framed prompts produce higher task completion rates than role-based prompts?
2. Do animistic-framed prompts produce better behavioral consistency (pass^k) across repeated trials?
3. Do animistic-framed prompts produce lower scope violation rates?
4. Are these effects consistent across different foundation models?

## Study Design

### Prompt Conditions

Three conditions applied to the same agent tasks:

**Condition A — Role-based, unstructured**
The baseline most practitioners actually use. "You are a [role]. Help users with [domain]." Minimal structure, no explicit constraints.

**Condition B — Role-based, structured (RISEN)**
A fully formatted RISEN prompt for the same task. Controls for structure: if animistic prompts outperform this condition, it's not just because they're more structured.

**Condition C — Animistic (character sheet → prompt)**
A character sheet is completed for each domain, then translated into a system prompt using a fixed template. The character sheet fields — artifact identity, what it protects, what it remembers, what it resists, its blind spot, and a mood grid of behavioral triggers — drive the prompt content.

All prompts are written by the researchers (not generated by an LLM) to avoid confounding the prompt-generation method with the framing under test.

### Benchmark

**[τ²-bench](https://github.com/sierra-research/tau2-bench)** (Sierra Research, ICLR 2025)

τ²-bench simulates conversations between an LLM-simulated user and an agent equipped with domain-specific tools and policy guidelines. It evaluates whether the agent completes tasks while adhering to domain policies — a direct test of the scope-adherence and consistency claims central to this study.

Domains:
- **Retail** (~115 tasks): e-commerce customer service with product, order, and return policies
- **Airline** (~50 tasks): booking and support with fare rules and ticket policies

### Models

- GPT-4o (OpenAI)
- Claude Sonnet (Anthropic)

### Metrics

- **pass@1**: task completion rate across all tasks
- **pass^k**: fraction of tasks where all k repeated trials succeed (behavioral consistency)
- **DB reward**: policy adherence — whether the agent's actions left the database in the correct state
- **Scope violation rate**: actions taken outside the agent's defined scope

## Hypotheses

- **H1**: Condition C ≥ Condition B > Condition A on policy adherence and pass^k
- **H2**: Condition A may match or exceed C on raw task completion — animistic prompts are more restrictive by design
- **H3**: Effects are consistent across GPT-4o and Claude Sonnet
- **H4**: Failures in Condition C transcripts are more attributable to specific prompt rules than in A or B

## Setup

**Prerequisites**: [uv](https://github.com/astral-sh/uv) and API keys for OpenAI and/or Anthropic.

```bash
# 1. Clone this repo and τ²-bench side by side
git clone https://github.com/chrizbo/animistic-agents-research
cd animistic-agents-research
git clone https://github.com/sierra-research/tau2-bench
cd tau2-bench && uv sync && cd ..

# 2. Configure API keys
cp .env.example .env
# edit .env and add your keys

# 3. Register the custom agent with τ²-bench
./scripts/setup_tau2.sh
```

## Running Evaluations

```bash
# Validate the pipeline (1 task only)
./scripts/run_eval.sh --condition c --domain retail --model gpt-4o --test

# Full k=1 run for one condition
./scripts/run_eval.sh --condition a --domain retail --model gpt-4o
./scripts/run_eval.sh --condition b --domain retail --model gpt-4o
./scripts/run_eval.sh --condition c --domain retail --model gpt-4o

# Options
#   --condition   a | b | c | cv2
#   --domain      retail | airline
#   --model       gpt-4o | claude-sonnet-4-5
#   --user-model  model for the simulated user (default: gpt-4o)
#   --trials      number of trials per task (default: 1; use 5 for final runs)
#   --num-tasks   run a subset of tasks (e.g. 20 for a cheap pilot)
#   --concurrency max parallel simulations (default: 5)
```

## Analysis

```bash
uv run python3 scripts/analyze.py --results-dir results/ --output-dir analysis/
open analysis/pass_at_1_retail.png
```

Produces bar charts (PNG + PDF) comparing pass@1 and DB reward across conditions and models, plus a `summary.csv`.
