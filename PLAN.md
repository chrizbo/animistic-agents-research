# Animistic Agent Design: Empirical Study Plan

## Overview

This study empirically tests whether system prompts derived from animistic agent framing (treating agents as artifacts rather than roles) produce better-performing agents than role-based prompts, as measured on a standard agent evaluation benchmark.

The central claim under test: artifact-framed agents are more consistent, more scope-adherent, and produce more legible failures than role-framed agents — not just in theory, but measurably, on real tasks.

---

## Background

Role-based agent design (e.g. "You are a customer support agent") is the dominant paradigm in LLM deployment. Structured templates like RISEN (Role, Instructions, Steps, Expectation, Narrowing) represent its most disciplined form. The animistic alternative treats an agent as an artifact with defined properties, boundaries, and failure modes — specified using a character sheet that maps the agent's identity onto a concrete object (e.g. a washing machine, a compliance record).

Prior evidence is anecdotal: a Foo Camp 2026 session found that the character sheet produced tighter, more intuitive agent specs than role framing, including from non-technical participants. This study generates quantitative evidence.

---

## Research Questions

1. Do animistic-framed system prompts produce higher task completion rates than role-based prompts?
2. Do animistic-framed prompts produce better behavioral consistency (pass^k) across repeated trials?
3. Do animistic-framed prompts produce lower scope violation rates?
4. Are these effects consistent across different foundation models?

---

## Study Design

### Conditions

Three prompt conditions, applied to the same agent tasks:

**Condition A — Role-based, unstructured**
The baseline most practitioners actually use. Format: "You are a [role]. Help users with [domain]." Minimal structure, no explicit constraints.

**Condition B — Role-based, structured (RISEN)**
A properly formatted RISEN prompt (Role, Instructions, Steps, Expectation, Narrowing) for the same task. Controls for structure: if animistic prompts outperform this condition, it's not just because they're more structured.

**Condition C — Animistic (character sheet → prompt)**
The character sheet is completed for each agent task domain, then translated into a system prompt following a consistent procedure. The character sheet fields (artifact identity, what it protects, what it remembers, what it resists, its blind spot) drive the prompt content. The resulting prompt — not the sheet itself — is what gets evaluated.

All prompts are written by the researchers (not generated by an LLM) to avoid confounding the prompt-generation method with the framing under test.

### Domains

Both τ³-bench domains:
- **Retail** (~115 tasks): e-commerce customer service with product, order, and return policies
- **Airline** (~50 tasks): booking and support with fare rules and ticket policies

Running both domains strengthens generalizability claims across task types.

### Models

Two foundation models tested in each condition:
- **GPT-4o** (OpenAI)
- **Claude Sonnet** (Anthropic)

Using two models tests whether observed effects are model-specific or reflect properties of the prompting approach itself.

### Trial configuration

Configurable at run time via `--num-trials` flag (default: 1).

- **k=1**: Single trial per task. Use for pilot runs and early signal checking. Fast and cheap (~$40/domain/model).
- **k=5**: Five repeated trials per task. Use for final results. Enables pass^k reliability metric. (~$200/domain/model)

Recommended sequence: run k=1 across all conditions and models first to validate setup and check for signal, then commit to k=5 for final results.

**Estimated total cost (k=5, all conditions, both domains, both models):**
~$1,000–$1,500 depending on model pricing.

### Test mode

A `--test` flag runs a single task per domain rather than the full set. Use this during initial setup to confirm the pipeline is wired correctly before any significant API spend.

```bash
python run.py --test ...   # runs 1 task only
```

Test mode does not produce meaningful metrics — it is purely for code validation.

## Evaluation Benchmark

**τ³-bench** (Sierra Research)
Repository: https://github.com/sierra-research/tau2-bench

τ³-bench simulates conversations between a user (LLM-simulated) and an agent equipped with domain-specific tools and policy guidelines. It evaluates whether the agent completes tasks while adhering to domain policies — a direct test of the scope-adherence and consistency claims central to this study.

τ³-bench was chosen over alternatives (SOPBench, AGENTIF, AgentBench) because:
- It directly measures policy adherence, not just task completion
- It has the pass^k reliability metric built in
- Its domains (retail, airline) have natural scope boundaries well-suited to artifact framing
- It is a published, peer-reviewed benchmark (ICLR 2025)

---

## Metrics

**Primary:**
- **Task completion rate** (pass@1): did the agent complete the task correctly?
- **Behavioral consistency** (pass^k): probability of succeeding on all k repeated attempts of the same task
- **Policy adherence rate**: did the agent follow domain-specific rules and constraints?

**Secondary:**
- **Scope violation rate**: did the agent take actions outside its defined scope?
- **Failure attribution** (qualitative): for a sample of failures, can a blind rater identify the cause from the prompt and transcript alone? Scored on a 3-point scale: fully attributable / partially attributable / not attributable.

---

## Hypotheses

- **H1**: Condition C (animistic) ≥ Condition B (RISEN) > Condition A (unstructured) on policy adherence and pass^k.
- **H2**: Condition A may match or exceed C on raw task completion — animistic prompts may be more restrictive by design. This is expected and should be reported honestly.
- **H3**: Effects are consistent across GPT-4o and Claude Sonnet, suggesting they reflect prompting approach rather than model-specific behavior.
- **H4**: Failure attribution scores are higher for Condition C transcripts than A or B.

## Character Sheet → Prompt Translation Template

The animistic prompt (Condition C) is always derived from a completed character sheet using this fixed translation template. Using a fixed template ensures reproducibility: any researcher with the same character sheet should produce an equivalent prompt.

### Character sheet fields

| Field | Prompt for author |
|---|---|
| **Artifact name** | What object is this agent? (e.g. "a return processing machine") |
| **What I protect** | What is the core thing this agent must preserve or guarantee? |
| **What I remember** | What inputs and state does this agent track? |
| **What I resist** | What requests or pressures should this agent refuse or ignore? |
| **My blind spot** | What is explicitly outside this agent's awareness or responsibility? |
| **Mood grid** | (Optional) emotional/tonal register across situations |

### Translation template

```
You are [artifact name].

Your purpose is to [what I protect]. You do not make exceptions to this.

You work with: [what I remember]. Everything you do is grounded in these inputs.

You do not [what I resist]. If asked to do so, you decline and explain why it falls
outside your function.

You are not responsible for [my blind spot]. When these topics arise, you
acknowledge them and redirect to whoever or whatever is responsible.
```

The translation should be done by the researchers, not an LLM, for the same reason the prompts are researcher-written: to avoid confounding framing with generation method.

## Run Procedure

### Setup

```bash
# Clone tau3-bench (uses uv, not pip)
git clone https://github.com/sierra-research/tau2-bench
cd tau2-bench
uv sync

# Set API keys (copy sample and fill in)
cp ../animistic-agents-research/.env.example .env
# edit .env with your keys
```

### Running an evaluation

```bash
# General form
tau2 run \
  --domain retail \                        # or: airline
  --agent-llm gpt-4o \                     # or: claude-sonnet-4-5
  --user-llm gpt-4o \
  --num-trials 1 \                         # set to 5 for final runs
  --num-tasks 115 \                        # full retail set; 50 for airline
  --agent-system-prompt ../animistic-agents-research/prompts/retail/condition-c-animistic.md
```

**Test mode** — single task, purely for validating the pipeline:

```bash
tau2 run --domain retail --agent-llm gpt-4o --user-llm gpt-4o \
  --num-trials 1 --num-tasks 1 \
  --agent-system-prompt prompts/retail/condition-c-animistic.md
```

Results are saved to `tau2-bench/data/simulations/`. Copy completed runs to `animistic-agents-research/results/` and name them by condition, domain, model, and trial count for traceability (e.g. `retail_condC_gpt4o_k1_2026-07-01.json`).

### Generating graphs

After one or more runs are complete, run the analysis script to generate comparison plots:

```bash
python scripts/analyze.py --results-dir results/ --output-dir analysis/
```

This produces:
- Bar charts comparing task completion rate (pass@1) across conditions A/B/C, per domain and model
- Bar charts comparing policy adherence rate across conditions
- pass^k consistency plot (when k>1 results are available)
- A summary table (CSV) of all metrics across all runs

All charts are saved as PNG and PDF to `analysis/` for direct inclusion in the paper.

## Design Notes

### On compound artifacts

During character sheet development for the airline domain, a natural question arose: should an agent be allowed to have *two* artifacts that work together, rather than one? The ticket machine and flight manifest are in a producer-consumer relationship — the machine acts, the manifest records. Forcing a single artifact risks flattening a genuinely two-part identity.

For this study, we use the simpler pattern: the agent *is* the ticket machine, and the manifest is *what it protects* (the "what I protect" field). This keeps the character sheet structure clean and the prompt translation reproducible.

However, the compound artifact pattern — where an agent's identity is expressed as a relationship between two artifacts — is a plausible extension of the framework for agents with separable concerns (e.g. a processing component and a record component). This is worth exploring in future work, and the airline domain is a motivating example.

## Open Questions

- **IRB**: No review needed for this study — there are no human subjects. The only humans involved are the researchers writing prompts, which is normal research activity, not human subjects research. The simulated users in τ³-bench are LLMs. That said, if the paper eventually includes a human rater component for failure attribution (e.g. recruiting external annotators via Prolific), that portion would warrant an IRB exemption request, which is straightforward for low-risk annotation tasks.
